# State Machine — Incident (Инцидент/Тревога)

## 1. Сущность
Incident = событие SOS или обращение, которое проходит жизненный цикл от создания до закрытия.

## 2. Статусы
| Код | Название | Цвет UI | Кто видит |
|---|---|---|---|
| NEW | Новая | Красный | Все в пульте |
| ACKED | Принята системой (ACK) | Красный | Оператор |
| DISPATCHED | Назначена силам | Оранжевый | Все |
| ACCEPTED | Принята охранником/экипажем | Синий | Все |
| EN_ROUTE | В пути | Синий | Все |
| ON_SCENE | На месте | Фиолетовый/Синий | Все |
| RESOLVED | Завершена (успешно) | Зелёный | Все |
| CANCELED | Отменена клиентом/оператором | Серый | Все |
| FALSE_ALARM | Ложная тревога | Серый | Все |
| FAILED | Ошибка/недоставлено/технический сбой | Чёрный/Тёмный | Админ |

Примечание: `ACKED` — технический статус “сервер принял”. В UI можно не показывать отдельной строкой (по решению), но в истории хранить.

## 3. Переходы и правила
### 3.1 Автоматические
- NEW → ACKED (после успешного сохранения на сервере и выдачи ACK)
- Любой → FAILED (только системный переход при критической ошибке, фиксируется сервером)

### 3.2 Оператор
- ACKED/NEW → DISPATCHED (когда создал Dispatch)
  - Требует: хотя бы 1 получатель.
- DISPATCHED → ACCEPTED (если подтверждение пришло от охранника/экипажа через приложение)
  - Может происходить автоматически.
- Любой (кроме RESOLVED) → CANCELED
  - Требует: **Комментарий обязателен**.
- Любой (кроме RESOLVED) → FALSE_ALARM
  - Требует: **Комментарий обязателен**.

### 3.3 Охранник/Экипаж
- DISPATCHED → ACCEPTED (кнопка “Принять”)
- ACCEPTED → EN_ROUTE
- EN_ROUTE → ON_SCENE
- ON_SCENE → RESOLVED
  - Требует: итоговый комментарий/отчёт (минимум 1 строка).

### 3.4 Админ/Суперадмин
- Может переводить в любой статус (кроме нарушающих логику по запрету ниже), но:
  - **Любая ручная смена статуса требует комментарий**.
  - Должна фиксироваться в истории с причиной.

## 4. Запреты
- RESOLVED — финальный: обратно нельзя (только суперадмин может создать новый инцидент, но не “откатывать”).
- Нельзя ставить DISPATCHED без Dispatch.
- Нельзя ставить ACCEPTED без указания принявшего (guardId/patrolUnitId).

## 5. Обязательные поля при событиях
### 5.1 При создании Incident
- clientId / userId
- координаты (lat/lon) если доступны
- deviceTime
- source = CLIENT_SOS / OPERATOR_MANUAL (на будущее)

### 5.2 При создании Dispatch
- incidentId
- recipients[] (посты/экипажи/охранники) минимум 1
- method = RADIO / PHONE / APP / MIXED
- если RADIO: acceptedBy (строка) + acceptedAt (время), может быть позже

### 5.3 При любом ручном изменении статуса
- comment обязателен
- actor (кто сделал)

## 6. История статусов (audit)
На каждый переход пишем запись:
- incidentId
- fromStatus, toStatus
- actorUserId + actorRole
- comment (если обязателен) / systemReason
- createdAt (UTC)

## 7. SignalR события
- IncidentCreated (NEW/ACKED)
- IncidentStatusChanged (любой переход)
- DispatchCreated (назначение)
- DispatchAccepted (принятие охранником)
- GuardLocationUpdated (для активного трекинга)
