# PRD — Система тревожной кнопки и диспетчеризации для ЧОП

## 1. Цель продукта
Сделать единую систему:
- **Мобильное приложение (MAUI Blazor Hybrid)** для **Клиента** и **Охранника**.
- **Веб-пульт (Blazor Web App)** для **Оператора / Админа / Кадровика / Суперадмина**.
- **Backend** (ASP.NET Core API + SignalR) и **PostgreSQL + PostGIS**.

Ключевая ценность: **SOS-тревога с геолокацией** + **быстрая диспетчеризация** ближайшим силам + **прозрачная история**.

## 2. Роли и права (RBAC)
### Клиент (Mobile)
- SOS (создать инцидент, передать геолокацию).
- Профиль, несколько адресов.
- Статус обслуживания/оплаты (read-only).
- Контакты ЧОП.
- Восстановление доступа (если вышел).

### Охранник (Mobile)
- “Заступил на пост” / “Закончил смену”.
- Пинги геолокации: обычный режим и режим активной заявки.
- Принять заявку / менять статусы (только разрешённые переходы) / комментарии.
- Видеть назначенные ему заявки.

### Оператор (Web)
- Получать новые тревоги (realtime).
- Видеть карточку клиента (адреса, телефоны) + геоточку.
- Видеть ближайшие силы (посты и экипажи) + контакты + ответственных + частоты раций.
- Создавать диспетчеризацию (назначить 1..N получателей).
- Менять статусы заявки (с обязательным комментарием при ручной смене).
- Смотреть “силы на смене” и их статусы.

### Кадровик (Web)
- Управление охранниками (создание, приглашения, блокировки).
- Управление сменами и назначениями (кто где стоит).
- Нет доступа к платежам.

### Администратор (Web)
- Всё оператора + клиенты, договоры/тарифы, платежи, импорт выписок.
- Управление объектами, постами, экипажами, справочниками.

### Суперадмин (Web)
- Полный доступ + системные настройки + аварийная смена паролей/доступов.
- Просмотр расширенного аудита.

## 3. Основные сценарии (User Journeys)
### 3.1 SOS клиента
1) Клиент нажимает SOS → приложение получает координаты.
2) Отправляет инцидент в API → API подтверждает принятие (ACK).
3) Оператор мгновенно видит инцидент в пульте (SignalR).
4) Оператор назначает ближайшие силы → заявка меняет статус.
5) Охранник принимает → трекинг становится чаще → закрытие с итогом.

### 3.2 Дежурство охранника
1) Охранник “Заступил” (выбор поста/назначение).
2) Пинги раз в 10 минут (обычно).
3) При назначении заявки → режим 10 сек.
4) При закрытии заявки → вернуться к обычному режиму.
5) Завершить смену.

### 3.3 Импорт оплат
1) Админ загружает TXT-выписку `1CClientBankExchange`.
2) Система парсит строки платежей, сохраняет “как есть” + нормализованные поля.
3) Пытается сопоставить оплату клиенту.
4) Автосопоставленные → применяются. Спорные → очередь ручного разбора.

## 4. MVP (первый релиз)
### MVP-1 (минимум работоспособности)
- Авторизация + роли.
- Приглашения (клиенты/охранники) + первый вход = смена пароля.
- SOS клиента → инцидент в БД → realtime в веб-пульт.
- Пульт оператора: список инцидентов + карточка + смена статусов + комментарии.
- Справочник постов/экипажей с координатами.
- “Ближайшие силы” (PostGIS) по координатам инцидента.
- Диспетчеризация: назначить 1..N получателей + метод связи + “кто принял по рации”.
- Охранник: “заступил” + пинги локации (10 мин) + “принять заявку” + режим 10 сек.
- Аудит статусов инцидента (кто/когда/почему).

### MVP-2 (финансы)
- Клиенты/договоры/период обслуживания.
- Статус оплаты в мобильном кабинете клиента.
- Импорт `1CClientBankExchange` + автосопоставление + ручная очередь.

## 5. Next (после MVP)
- Push-уведомления (операторам/охранникам).
- Геозоны для постов, алерты “вышел из зоны”.
- Расчёт ETA (примерный) для ближайших сил.
- Отчёты по SLA/реакции/нагрузке.
- 2FA для админов.
- Мульти-организация (несколько ЧОП в одной системе) — опционально.

## 6. Нефункциональные требования
- Realtime доставка инцидента в веб-пульт: цель ≤ 5 сек при нормальной сети.
- Надёжность: оффлайн-очередь SOS и ретраи до ACK.
- Аудит: все изменения статусов/назначений — в истории.
- Безопасность: токены, ограничение прав, rate limiting на критичные операции.
- Логи: централизованные, без персональных данных в логах по умолчанию.

## 7. Ограничения и допущения
- В MVP: один мобильный апп с разными режимами UI для Клиента/Охранника (по роли).
- Геолокация в фоне ограничена правилами iOS/Android; в MVP делаем корректно, но без “хака” ОС.
