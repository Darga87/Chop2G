# Payments — Импорт TXT (1CClientBankExchange)

## 0. Принцип “без фантазий”
Мы НЕ завязываемся на “один единственный набор полей”.
Формат 1CClientBankExchange — текстовый, ключ=значение, с секциями.
Разные банки/клиент-банки могут добавлять/менять набор ключей.

Поэтому:
1) Парсер должен быть **толерантным**: всё неизвестное сохраняем в `extra (jsonb)`.
2) Нормализуем только те поля, которые реально присутствуют.
3) Для сопоставления используем набор извлечённых значений (дата, сумма, назначение, счета, ИНН…).

## 1. Поток импорта
1) Upload файла → создаём bank_imports
2) Parse → bank_import_rows
3) Auto-match → заполняем match_candidate_client_ids + match_status
4) Apply auto-matched → создаём payments и связываем
5) Manual review → админ выбирает clientId → создаём/связываем payment

## 2. Парсинг (общие правила)
- Кодировка: читать как заявлено в заголовке, если есть; иначе UTF-8 с fallback.
- Строки формата:
  - `Ключ=Значение`
  - секции начинаются/заканчиваются спец-ключами (например, `СекцияДокумент=...`, `КонецДокумента`, и т.п.)
- Для каждой “документ-секции” создаём `bank_import_rows`.
- В `extra` сохраняем:
  - все ключи (включая дубли, если встретились — хранить как массив или последнее значение; решение фиксируем в DECISIONS)

## 3. Нормализация в bank_import_rows
Нормализуем только то, что нашли:
- doc_type (тип документа/секции)
- doc_no
- doc_date
- amount
- payer_name
- payer_inn
- payer_account
- receiver_account
- purpose

Остальное — `extra`.

## 4. Алгоритм сопоставления (Auto-match)
Сопоставление делаем по приоритету:

### 4.1 Точное сопоставление (P1)
- В назначении платежа (`purpose`) найден идентификатор договора/счёта/клиента:
  - шаблоны: `ID:xxxxx`, `ДОГОВОР №`, `СЧЕТ №`
- Совпадает `contract_no` или другой ключ в clients.

### 4.2 Сопоставление по реквизитам (P2)
- payer_inn совпадает с client.inn (если ведём)
- payer_account совпадает с сохранённым payer_account клиента (если ведём)

### 4.3 Сопоставление по “слабым признакам” (P3)
- payer_name похож (fuzzy) + сумма совпадает + дата в допустимом окне

### 4.4 Текущий MVP (реализовано)
- Реализован P1 по `purpose`:
  - `ID:xxxxx`
  - `ДОГОВОР №xxxxx`
  - `СЧЕТ №xxxxx`
- В MVP ключ `xxxxx` сопоставляется с `client_profiles.user_id` (как “другой ключ в clients”).
- Если из `purpose` извлечено:
  - 1 валидный client key -> `MATCHED`
  - 2+ валидных client keys -> `AMBIGUOUS` (manual queue)
  - 0 валидных client keys -> `UNMATCHED`
- P2/P3 остаются в roadmap до появления в модели клиентов реквизитов (`inn`, `payer_account`, billing-period signals).

Правила:
- Если найден **ровно 1 кандидат** → AUTO.
- Если **2+ кандидата** → NEEDS_REVIEW.
- Если **0** → UNMATCHED.

## 5. Дубли, частичные оплаты, возвраты
- Дубль: если file_hash уже был → не применяем повторно (но можем показать предупреждение).
- Частичная оплата: допускаем несколько payments на один период (решение фиксируем в DECISIONS).
- Возвраты/отрицательные суммы: в MVP можно помечать UNMATCHED и разбирать вручную.

## 6. Ручной разбор (Manual review)
UI показывает строку и кандидатов:
- purpose, payer_name, сумма, дата, счета, extra
Админ выбирает clientId → создаём payment, связываем row → status MANUAL.
